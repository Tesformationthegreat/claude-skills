# Design system reference

This file is the line between an app that looks like it was shipped by a product team and an app that looks like it was generated by an AI in a hurry. Load this before writing any UI code.

## The core rule

Do not write a single line of UI code until a design direction is locked. That means concrete answers to:

1. **Product tone.** Serious, playful, technical, warm, premium, utilitarian?
2. **Target user.** Who opens this app and what are they feeling when they do?
3. **Visual references.** Real products the UI should feel like. Not "clean and modern". Actual names.
4. **Typography.** Which fonts, which weights, for what.
5. **Color system.** A restrained palette. Not "blue and some other colors".
6. **Spacing rhythm.** Base unit and scale.
7. **Component language.** Are corners sharp or soft? Are surfaces flat or layered? Are borders visible?
8. **Mobile behavior.** How the layout reflows, what sticks, what scrolls.

If the user has not given you these, ask. Propose defaults if they do not know. Get confirmation before writing code.

## What to avoid

These are the patterns that make an app look AI generated. Catch yourself before doing any of them.

**Generic gradients.** Blue to purple, pink to orange, teal to blue. If the gradient is decorative (not communicating data or state), it is probably wrong. When in doubt, solid color.

**Decorative blobs.** Soft blurred shapes floating in the background with no purpose. They read as "I did not know what else to put there".

**Overused dark mode with neon accents.** Black background, one bright accent color, glow effects. This is the default AI dashboard look. Only use it if the product genuinely calls for it (dev tools, music, gaming, nightlife).

**Samey SaaS feature grids.** Three columns, icon on top, heading, two lines of body text. Every landing page has this. If you must use it, make one element unexpected.

**Template layouts.** Hero with a big heading, subheading, two buttons, then three feature cards, then a testimonial, then a CTA. If this is the layout, there is no layout. Break it.

**Floating cards on gradient backgrounds.** Card with a shadow floating on a pastel gradient. Reads as stock template immediately.

**Rounded corners on everything with a drop shadow on everything.** Pick a radius and stick to it. Use shadows sparingly and with intent.

**Generic copy.** "Welcome to [product]. Get started today." "Unlock your potential." "Transform your workflow." If the copy could apply to any product, it is not copy.

**Cluttered dashboards.** Six charts, four number cards, a table, a filter bar, and a sidebar on the first view. Show one thing well, not ten things poorly.

**Emoji in every heading.** ⚡ Fast. 🚀 Scalable. 🎯 Focused. This is lazy. Use emoji intentionally or not at all.

**Lucide icons in every card without meaning.** Every feature card with a different Lucide icon picked by vibes. Icons should communicate, not decorate.

## What to do instead

**Premium hierarchy.** One thing is the biggest. One thing is the most saturated. One thing is the most prominent. The rest recedes. If three things are fighting to be the most important, nothing is important.

**Restrained color.** One primary, one accent, a neutral scale (5 to 9 steps of gray), and semantic colors (success, warning, error). That is it. Do not add more colors because the page looks empty. The page looks empty because of spacing or typography, not color.

**Elegant typography.** Pick two families at most, often just one. A serif or display font for headings, a system font or a clean sans for body, sometimes the same family for both. Use weight and size for hierarchy, not color.

Good pairings:
- Inter for everything (safe default, almost always fine)
- Geist Sans for UI, Geist Mono for code
- Source Serif for headings, Inter for body (editorial feel)
- IBM Plex Sans plus IBM Plex Mono (technical, credible)
- Fraunces for display, Inter for body (warm, premium)

Bad pairings:
- Three different Google Fonts because they looked nice individually
- Montserrat for everything (overused, feels templated)
- Poppins for anything serious (reads as startup cliché in 2026)

**Spacing discipline.** Pick a base unit (4px or 8px) and scale from there. Consistent spacing is what makes an interface feel designed.

Default scale: 4, 8, 12, 16, 24, 32, 48, 64, 96, 128.

**Component language.** Pick a vocabulary and stick to it.
- Radius: 0, 4, or 8. Not 12 on some things and 6 on others.
- Borders: either every card has one or none do.
- Surfaces: either elevation with shadow or elevation with border. Not both on the same layer.
- Buttons: two variants, not seven. Primary and secondary, and maybe ghost.

**Mobile first.** Design the mobile layout first. If it works on a 375px screen, it will scale up. If it only works on desktop, it will never feel right on mobile.

**States matter.** Every interactive thing has:
- Default state
- Hover state (desktop)
- Active/pressed state
- Focus state (keyboard accessibility)
- Disabled state
- Loading state

Every data view has:
- Loading state (skeleton, not spinner, usually)
- Empty state (not "No data". Give them something to do.)
- Error state (specific, actionable)
- Success state (where applicable)

Every form has:
- Field level validation
- Submission loading state
- Submission success feedback
- Submission error handling

Missing states are the biggest tell of an unfinished product.

**Subtle motion.** Motion is for feedback, not decoration. Things that appear should fade or slide in over 150 to 250 ms. Things that change should transition over 100 to 200 ms. Never animate for longer than 400 ms on a UI interaction. Reduce motion respected via `prefers-reduced-motion`.

## Accessibility floor

Non negotiable:
- Color contrast meets WCAG AA (4.5:1 for body text, 3:1 for large text and UI)
- Every interactive element reachable by keyboard
- Focus states visible
- Form inputs have labels (not just placeholders)
- Images have alt text (or empty alt if decorative)
- Motion can be disabled
- Semantic HTML (use a button for a button, not a div with an onClick)

## Responsive floor

Test at these breakpoints minimum:
- 375px (iPhone SE, iPhone mini)
- 768px (iPad portrait)
- 1280px (common laptop)
- 1920px (desktop)

A layout that works at 320px to 2560px is the goal. If text is cut off on iPhone SE, the layout is broken.

## Copy rules

- Write like a human, not a brand
- Specific beats generic. "Stripe style checkout in 10 minutes" beats "Streamline your payments"
- No filler (unlock, elevate, transform, seamless, cutting edge, next gen, revolutionary)
- Sentence case for headings (not Title Case For Everything)
- Active voice
- Short sentences

## Before writing UI code, ask these

If the user has not answered, ask. If they do not know, propose and confirm.

1. What three real products should this feel like?
2. What is the primary action on the main screen?
3. Is this a tool people use for hours or minutes?
4. Is it mobile first, desktop first, or both equally?
5. What is the emotional tone in one word?
6. Any hard constraints (brand colors, existing design system)?

Once you have these, lock a direction and build from it. Do not drift halfway through.

## The taste test

Before shipping any UI, ask:
- Would this look out of place on a page of real production apps?
- Does any element feel decorative rather than functional?
- Is there a single visual focal point on the main screen?
- Does the copy sound like a human wrote it?
- Does the spacing feel intentional?
- Are there any gradients, blobs, or neon accents that do not need to be there?
- Would a designer from a company with taste (Linear, Stripe, Vercel, Notion, Raycast, Arc) ship this?

If any answer is no or "I am not sure", keep working. This is the difference between shipped and shippable.

## Premium reference set

When you need a concrete "make it feel like this" reference, use these. Do not copy, but study the patterns.

- **Linear** for dense, keyboard driven product UI
- **Stripe** for marketing plus technical product polish
- **Vercel** for developer tools with restraint
- **Notion** for content dense apps with soft edges
- **Arc** for playful but purposeful browser UI
- **Raycast** for extension style density
- **Apple** for consumer warmth and hierarchy
- **Bloomberg Businessweek** for editorial boldness if the product leans content
- **Monocle** for refined typography and layout
- **Things 3** for minimalist mobile with real craft

Do not use Dribbble as a reference. Dribbble is where generic AI aesthetics come from. Use real shipped products.
